# windPredictor

A sailing forecast system for local lake conditions.  A Random Forest model
trained on weather-station history predicts whether the sailing window will be
good.  NWP data from Open-Meteo enriches the output with expected wind/gust/
direction-consistency values.  The result is a self-contained `index.html`
deployed to GitHub Pages via Actions artifact on every scheduled run.

---

## Architecture

```
input/
  scraper.py        Download daily xlsx files from Ecowitt weather station
  stitcher.py       Merge xlsx → weather_readings table (Supabase or local.db)
  weather_store.py  load_weather_readings() / upsert_readings() — weather DB layer
  open_meteo.py     Fetch NWP forecast (Open-Meteo, no API key needed)

model/
  features.py       Extract 28 features + compute daily sailing-quality targets
  train.py          Train Random Forest → model/weights.joblib
  predict.py        predict_now() — predict from current time; save to forecast_snapshots table
  history.py        Persist direct ML predictions + observed outcomes to predictions/outcomes tables

utils/
  circular.py       circular_std(angles) — single implementation, used by model and render
  config.py         load_config(path) — loads config.toml + calls load_dotenv()
  db.py             get_connection(), backend(), placeholder(), DEFAULT_SQLITE

render/
  charts.py         prob_trend_svg(), wind_svg()
  data.py           window_stats(), expected_wind_chips(), stats_html(), history_html()

render_html.py      Render self-contained index.html from forecast_snapshots + history DB
deploy.py           Local pipeline: download → stitch → predict → render [→ preview]

supabase/
  schema.sql             Core predictions/outcomes tables
  schema_additions.sql   weather_readings + forecast_snapshots tables
  migrate_parquet.py     One-time: load data.parquet → weather_readings
  migrate_from_sqlite.py One-time: push predictions.db → Supabase
```

---

## Mutable state — where it lives

| Data | Supabase table | SQLite fallback (local.db) |
|------|----------------|---------------------------|
| Weather station time series | `weather_readings` | same schema |
| Rolling 7-day forecast window | `forecast_snapshots` | same schema |
| ML prediction history | `predictions` | same schema |
| Observed outcomes | `outcomes` | same schema |

**Git contains only code and config.** `data.parquet`, `predictions.json`, and
`index.html` are all gitignored.  GitHub Pages is deployed via
`actions/upload-pages-artifact` + `actions/deploy-pages` — the CI pipeline
never commits any files.

---

## How a prediction run works

`python model/predict.py` (called by GitHub Actions and `deploy.py`):

1. Runs `predict_now()` at the current hour.
2. Loads the last 35 days of weather readings from `weather_readings`.
3. The direct ML prediction targets **today** (if before window_end 16:00) or
   **tomorrow** (if at or after window_end).  During the sailing window the
   features incorporate live station readings up to the current moment.
4. When the target is tomorrow, today's completed window data is surfaced as a
   separate **observed-only** entry (not stored to DB).
5. Extended forecasts for the remaining days up to **day+3** are generated by
   scaling the direct probability with decay factors (×1.0, ×0.82, ×0.64 for
   days +1, +2, +3 beyond the direct target).
6. Results are saved via `save_forecast_snapshots()` into `forecast_snapshots`
   (rolling 7-day window, deduplicated by snapshot + predicting_date; entries
   older than 7 days are pruned automatically).
7. Direct predictions (not extended or observed-only) are written to the
   `predictions` table for accuracy tracking.
8. `render_html.py` reads from `forecast_snapshots` via `load_forecast_snapshots()`
   and from the history tables to produce `index.html`.

## Scheduling (GitHub Actions)

Six runs per day via `.github/workflows/forecast.yml`:

| Cron (UTC) | CET  | CEST |
|-----------|------|------|
| `0 4 * * *`  | 05:00 | 06:00 |
| `0 6 * * *`  | 07:00 | 08:00 |
| `0 9 * * *`  | 10:00 | 11:00 |
| `0 12 * * *` | 13:00 | 14:00 |
| `0 17 * * *` | 18:00 | 19:00 |
| `0 21 * * *` | 22:00 | 23:00 |

The CI workflow deploys `index.html` to GitHub Pages via artifact — **no git
commits are ever made by the pipeline**.

## Configuration (`config.toml`)

| Section | Key | Description |
|---------|-----|-------------|
| `[sailing]` | `window_start/end` | Sailing window times (local) |
| `[sailing]` | `wind_speed_min/max` | Speed thresholds in knots |
| `[sailing]` | `wind_dir_consistency_max` | Max circular std of direction (°) |
| `[sailing]` | `min_temperature` | Minimum air temp (°C) to count as "good" |
| `[prediction]` | `min_good_fraction` | Fraction of window that must be good (default 0.30) |
| `[location]` | `lat`, `lon` | Enables Open-Meteo NWP enrichment |
| `[model]` | `n_estimators`, `max_depth` | Random Forest hyperparameters |
| `[ecowitt]` | `device_id`, `authorize` | Station credentials (override with env vars) |

## Credentials & Environment

**Local (`.env`, gitignored):**
```
SUPABASE_DB_URL=postgresql://...
ECOWITT_DEVICE_ID=...
ECOWITT_AUTHORIZE=...
```

Copy `.env.example` to `.env` and fill in values.  When `SUPABASE_DB_URL` is
unset all four tables fall back to `local.db` (SQLite).

**GitHub Secrets:** `SUPABASE_DB_URL`, `ECOWITT_DEVICE_ID`, `ECOWITT_AUTHORIZE`
must be set in Settings → Secrets and variables → Actions.

**GitHub Pages source** must be set to **"GitHub Actions"** in Settings → Pages.

## Local development

```bash
# Full pipeline (download 2 days, stitch to DB, predict, render)
python deploy.py

# Open result in browser after rendering
python deploy.py --preview

# Skip download if xlsx files are already present
python deploy.py --no-download

# Predict and render only (use existing DB data)
python model/predict.py
python render_html.py

# Train model from scratch (needed on first run)
python model/train.py

# Run tests
python -m pytest tests/ -v
```

### First-time setup
1. Copy `.env.example` to `.env` and fill in `SUPABASE_DB_URL` and Ecowitt credentials.
2. Run `python supabase/migrate_parquet.py` if you have an existing `data.parquet` to migrate.
3. Run `python deploy.py --days 90` to bootstrap 90 days of history and train the model.

## Key design decisions

- **All mutable state in Supabase** — weather readings, forecast snapshots, and
  prediction history all live in Supabase (SQLite fallback for local dev).  The
  repo contains only code and config; the pipeline never commits files.
- **GitHub Pages via artifact** — `actions/upload-pages-artifact` + `actions/deploy-pages`
  deploy `index.html` without any git commits, eliminating conflicts between
  local and CI runs.
- **No snapshot list in config** — each workflow run predicts at its current
  time.  Multiple runs per day accumulate in `forecast_snapshots`, building the
  probability-trend sparkline naturally.
- **Extended forecasts are display-only** — day+2/+3 entries are not stored
  in `predictions` to keep accuracy metrics clean.
- **Rolling 7-day forecast window** — old entries are pruned automatically from
  `forecast_snapshots`, so the UI always shows the last 7 days.
- **`_target_date` cutoff is `window_end` (16:00)** — before 16:00 → predict today;
  at/after 16:00 → predict tomorrow.
